# ADR: Configuración de Compilación y Gestión de Dependencias en Monorepo (PNPM + NestJS)

## Contexto

En un entorno de monorepo gestionado con PNPM, la resolución de módulos y la compilación de TypeScript en sub-aplicaciones (como `apps/backend`) pueden fallar silenciosamente si las configuraciones de directorios no son explícitas o si se rompen los enlaces simbólicos (symlinks) del workspace. Esto resulta en la no generación de la carpeta `dist` y en errores `MODULE_NOT_FOUND` al intentar levantar el servidor.

## Decisiones Técnicas y Configuración Crítica

Para garantizar la estabilidad del compilador y la correcta integración con el workspace de PNPM, los archivos de configuración en `apps/backend` deben mantenerse estrictamente con la siguiente estructura:

### 1. tsconfig.json

Es mandatorio declarar el bloque `include` para evitar la "ceguera" del compilador. El módulo debe ser `commonjs` para compatibilidad nativa con NestJS.

{
"compilerOptions": {
"module": "commonjs",
"declaration": true,
"removeComments": true,
"emitDecoratorMetadata": true,
"experimentalDecorators": true,
"allowSyntheticDefaultImports": true,
"target": "es2021",
"sourceMap": true,
"outDir": "./dist",
"rootDir": "./src",
"incremental": true,
"skipLibCheck": true,
"strictNullChecks": false,
"noImplicitAny": false,
"strictBindCallApply": false,
"forceConsistentCasingInFileNames": false,
"noFallthroughCasesInSwitch": false
},
"include": [
"src/**/*"
]
}

### 2. nest-cli.json

Define la raíz del código fuente y asegura la limpieza del directorio de salida previo a cada compilación.

{
"$schema": "https://json.schemastore.org/nest-cli",
"collection": "@nestjs/schematics",
"sourceRoot": "src",
"compilerOptions": {
"deleteOutDir": true
}
}

### 3. tsconfig.build.json

Hereda la configuración base y excluye explícitamente directorios de pruebas y dependencias del build de producción.

{
"extends": "./tsconfig.json",
"exclude": [
"node_modules",
"test",
"dist",
"**/*spec.ts"
]
}

## Consideraciones y Flujo de Trabajo (Troubleshooting)

### Regla de Oro de Dependencias

Queda estrictamente prohibido ejecutar `npm install`, `pnpm install` o `pnpm add` dentro de la carpeta `apps/backend` o cualquier otra sub-aplicación.
Toda gestión de paquetes debe realizarse desde la raíz del proyecto para preservar los enlaces simbólicos del monorepo.

### Diagnóstico: La carpeta "dist" no se genera

Si al ejecutar el entorno de desarrollo el compilador arroja "0 errors" pero la carpeta `dist` no se crea, el flujo de recuperación es el siguiente:

1. Verificar aislamiento de dependencias: Revisar si existe una carpeta `node_modules` física dentro de `apps/backend`. Si existe, eliminarla inmediatamente.
2. Reconstruir enlaces simbólicos: Ir a la raíz del proyecto y ejecutar la instalación global del workspace.
   Comandos:
   cd /raiz-del-proyecto
   pnpm install
3. Permisos de binarios (PNPM v10+): Si las dependencias no compilan sus binarios nativos (ej. bcrypt, @nestjs/core), autorizar los scripts desde la raíz.
   Comando:
   pnpm approve-builds
4. Limpiar caché de compilación: En la carpeta de la aplicación, eliminar la caché antes de reiniciar.
   Comando (PowerShell):
   Remove-Item -Recurse -Force dist, tsconfig.build.tsbuildinfo -ErrorAction SilentlyContinue

Siguiendo este estándar, se garantiza que TypeScript identifique correctamente los archivos fuente y que NestJS resuelva los módulos apuntando a la raíz del monorepo.
