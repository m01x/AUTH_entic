# ADR_04: Layout de Dashboard con Sidebar Glassmorphism y Routing por Vistas

**Fecha**: 09 de Mayo, 2026
**Última actualización**: 09 de Mayo, 2026
**Estado**: Aceptado
**Proyecto**: AUTH_entic

---

## 1. Contexto

Tras completar la fase de autenticación (Login/Register) con estilo glassmorphism, se requiere construir la experiencia post-login: un dashboard protegido con navegación lateral que agrupe las vistas de Home (posts), mantenedor de usuarios y perfil de usuario.

La interfaz debe mantener consistencia visual con las pantallas de login (glassmorphism, tonos oscuros, bordes redondeados) y seguir el stack tecnológico confirmado en ADR_003: **shadcn/ui** + **TanStack Router** + **Tailwind CSS**.

---

## 2. Decisiones Técnicas

### 2.1 Estructura de rutas con TanStack Router

Se utiliza el sistema de rutas basado en archivos de TanStack Router. El layout del dashboard se define en `dashboard/route.tsx` como ruta padre, con `<Outlet />` para renderizar las vistas hijas:

```
routes/
├── dashboard/
│   ├── route.tsx          # Layout padre: sidebar + <Outlet />
│   ├── index.tsx          # Home: jumbotron + tabla posts + formulario
│   ├── users.tsx          # Mantenedor de usuarios (En Construcción)
│   └── profile.tsx        # Perfil de usuario (En Construcción)
```

### 2.2 Sidebar glassmorphism flotante

El sidebar se posiciona como un elemento **fixed** con margen y bordes redondeados para dar efecto de "tarjeta flotante", consistente con el estilo de las cards de login:

- `fixed left-3 top-3` con `h-[calc(100vh-1.5rem)]`
- `rounded-2xl` para bordes curvos
- Clase `glass-card` para el efecto glassmorphism (heredado de `index.css`)
- Logo Authentic en la cabecera, navegación con Home y Users, y menú desplegable del usuario en el footer

El área de contenido principal compensa el margen con `ml-[calc(15rem+1.5rem)]`.

#### 2.2.1 Corrección del fondo del sidebar (post-implementación)

En la primera implementación, el `backgroundImage` y el overlay con blur se aplicaban exclusivamente al `<main>` (área de contenido). El sidebar, al ser un elemento `fixed` posicionado fuera del flujo de `<main>`, quedaba sobre el `bg-slate-950` sólido del contenedor raíz.

Esto provocaba que el `glass-card` del sidebar (con `background: white 10% transparent`) reflejara un negro sólido en lugar del efecto glossy esperado, rompiendo la consistencia visual con las cards del contenido (que sí tenían la imagen con blur detrás).

**Corrección**: Se movió el `backgroundImage` y el overlay `bg-black/50 backdrop-blur-[2px]` desde `<main>` al contenedor exterior `<div className="flex min-h-screen relative">`, cubriendo así toda la viewport. El sidebar ahora comparte el mismo fondo que las cards del contenido, logrando un efecto glassmorphism uniforme en toda la interfaz.

### 2.3 Menú desplegable del usuario (UserMenu)

Implementado con shadcn `<DropdownMenu>`, `<Avatar>` y `<AvatarFallback>`. Contiene:

- **Mi Perfil** → navega a `/dashboard/profile`
- **Cerrar Sesión** → acción mock con toast de sonner

Ubicado en el fondo del sidebar, con avatar + nombre + rol visibles como trigger.

### 2.4 Componentes shadcn/ui instalados

Se añadieron los siguientes componentes vía `npx shadcn@latest add`:

| Componente | Uso |
|------------|-----|
| `table` | Tabla de posts en Home |
| `dropdown-menu` | Menú desplegable del usuario |
| `card` | Componente base (no usado directamente; ver 2.5) |
| `textarea` | Campo descripción en formulario de post |
| `avatar` | Thumbnail del usuario en sidebar |
| `separator` | Separadores visuales |
| `skeleton` | Estados de carga (futuro) |

### 2.5 Incompatibilidad shadcn Card + glass-card (Corrección)

Se detectó que el componente `<Card>` de shadcn aplica las clases `bg-card` y `text-card-foreground`. En Tailwind 4, las utilidades (`@layer utilities`) tienen **mayor precedencia en cascada** que los componentes (`@layer components`), lo que provocaba que `bg-card` (blanco sólido) sobrescribiera el fondo translúcido de `glass-card`.

**Solución**: No usar `<Card>` de shadcn en contextos glassmorphism. En su lugar, usar `<div className="glass-card ...">` con estructura plana (mismo patrón que `LoginCard.tsx`).

### 2.6 Hook useToast centralizado

Se creó `src/hooks/useToast.ts` como wrapper sobre `sonner` para estandarizar las notificaciones:

```typescript
export const useToast = () => ({
  success: (msg: string) => toast.success(msg),
  error: (msg: string) => toast.error(msg),
  info: (msg: string) => toast.info(msg),
  warning: (msg: string) => toast.warning(msg),
});
```

Se exportan también las funciones individuales (`successToast`, `errorToast`, etc.) para uso sin hook.

### 2.7 Datos mock

Mientras el backend no expone endpoints de posts ni sesión de usuario, se trabaja con datos mock en `src/lib/mockData.ts`:

- `mockUser`: usuario simulado con id, nombre, email, rol, avatar
- `mockPosts`: array de 5 posts con título, descripción, autor y fecha
- `formatDate()`: formateo de fechas en español (CL)
- `getUserInitials()`: extrae iniciales para el avatar fallback

### 2.8 Vistas "En Construcción"

Las páginas `/dashboard/users` y `/dashboard/profile` comparten un componente genérico `<ComingSoon>` que muestra:

- Icono `Construction` de lucide-react con fondo amarillo translúcido
- Título y descripción personalizables
- Vector decorativo estilizado

### 2.9 Responsive

Por decisión, esta iteración es **desktop-only**. El responsive para móvil se abordará en iteraciones futuras.

## 4. Consideraciones y Consecuencias

### Positivas
- Estilo visual consistente con login/register (mismo glassmorphism, mismas clases CSS)
- Sidebar comparte el mismo fondo con blur que el contenido tras corrección del layout
- Navegación clara con roles definidos para cada vista
- Hook `useToast` centralizado evita duplicación de llamadas a sonner
- Mock data desacoplado facilita migración futura a API real
- Componentes genéricos (`ComingSoon`) reutilizables en otras vistas

### Deuda técnica aceptada
- Sidebar solo navega a Home y Users; Profile solo accesible vía dropdown
- Formulario de post sin validación Zod ni integración con backend real
- Vistas Users y Profile en estado placeholder
- Sin responsive para viewports móviles
- `DashboardComponent.tsx` quedará obsoleto tras esta iteración
