# ADR_05: Formulario de Usuario con Glassmorphism — Tipado y Componentes Shadcn

**Fecha**: 10 de Mayo, 2026
**Última actualización**: 10 de Mayo, 2026
**Estado**: Aceptado
**Proyecto**: AUTH_entic

---

## 1. Contexto

Con la ruta `/dashboard/users` ya definida (ADR_04) y el backend NestJS exponiendo un CRUD de usuarios alineado a un `CreateUserDto`, se requiere construir el formulario de creación de usuarios en el frontend. Dicho formulario debe:

- Reflejar fielmente los campos definidos en la entidad `User` del backend.
- Mantener la identidad visual glassmorphism del proyecto, consistente con `LoginCard` y `PostForm`.
- Utilizar exclusivamente componentes de shadcn/ui, conforme al stack confirmado en ADR_03.
- No incorporar lógica de estado, validación ni envío en esta iteración — únicamente la interfaz de usuario estática.

El componente existente `DashboardUserForm.tsx` era un stub (`<input className="bg-amber-200">`) sin relación con el diseño del sistema.

## 2. Decisiones Técnicas

### 2.1 Definición centralizada de la interfaz `User` y el tipo `UserRole`

Se creó el archivo `apps/frontend/src/types/user.ts` con los tipos exportables que modelan los datos esperados por el backend:

```typescript
export interface User {
    firstName: string;
    lastName: string;
    birthDate: string;
    email: string;
    password: string;
    role?: UserRole;
}

export type UserRole = 'USER' | 'ADMIN';
```

**Fundamento**: La interfaz refleja el `CreateUserDto` del backend (NestJS + class-validator), importando solo los campos relevantes para el formulario de registro. El tipo `UserRole` está acotado a los dos valores del enum del backend (`UserRole.USER`, `UserRole.ADMIN`). Centralizar estos tipos en `src/types/` permite que cualquier componente del frontend los reutilice sin depender del módulo de formulario.

El campo `password` se mantiene como `string` (no `passwordHash`) porque representa el valor en texto plano que el usuario ingresa, antes del hashing en el backend. El campo `birthDate` se modela como `string` (ISO 8601) para mantener compatibilidad con el DTO de NestJS, donde se valida con `@IsDateString()` y TypeORM lo convierte a `Date`.

**Archivo modificado**: `apps/frontend/src/types/user.ts` (se agregaron las palabras clave `export`).

### 2.2 Instalación del componente Select de shadcn/ui

El conjunto de componentes shadcn instalados hasta ADR_04 no incluía un componente de selección desplegable, requerido para el campo `role` del formulario. Se ejecutó:

```bash
npx shadcn@latest add select --yes
```

El componente exporta: `Select`, `SelectContent`, `SelectGroup`, `SelectItem`, `SelectLabel`, `SelectScrollDownButton`, `SelectScrollUpButton`, `SelectSeparator`, `SelectTrigger`, `SelectValue`.

### 2.3 Construcción del formulario DashboardUserForm

Se reescribió completamente el componente `apps/frontend/src/components/dashboard/DashboardUserForm.tsx`, pasando de un stub de 11 líneas a un formulario glassy de 155 líneas. Las decisiones de diseño fueron:

**Estructura del componente:**

```tsx
<div className="flex items-center justify-center min-h-[calc(100vh-8rem)]">
  <div className="glass-card w-full max-w-[520px] rounded-2xl px-7 py-8 sm:px-9 sm:py-10 ...">
    {/* Encabezado */}
    <div className="flex flex-col items-center">
      <h1>Nuevo Usuario</h1>
      <p>Completa los datos del usuario</p>
    </div>
    {/* Campos del formulario */}
    <form className="mt-7 space-y-5">
      {/* 6 campos: firstName, lastName, birthDate, email, password, role */}
    </form>
  </div>
</div>
```

**Centrado del formulario**: El wrapper exterior utiliza `flex items-center justify-center min-h-[calc(100vh-8rem)]`. Esto centra vertical y horizontalmente el formulario dentro del `<main>` del layout de dashboard (el cual ya tiene `ml-[calc(15rem+1.5rem)] p-8`). La resta de `8rem` compensa el padding del main y la altura de la top bar, evitando scroll innecesario.

**Ancho máximo**: `max-w-[520px]`, mayor que los `420px` de `LoginCard` porque este formulario tiene el doble de campos y un Select que requiere más espacio horizontal.

**Animación de entrada**: Se aplicó la clase `animate-in fade-in zoom-in-95 duration-500`, idéntica a la usada en `LoginCard`, para mantener coherencia en las transiciones entre vistas.

**Campos del formulario — mapeo interfaz `User` → UI:**

| Campo | Tipo de input | Ícono (lucide-react) | Placeholder | Label |
|---|---|---|---|---|
| `firstName` | `text` | `UserPen` | "Juan" | Nombre(s) |
| `lastName` | `text` | `UserPen` | "Pérez" | Apellido(s) |
| `birthDate` | `date` | `Calendar` | — | Fecha de nacimiento |
| `email` | `email` | `Mail` | "dev@tutor.com" | Correo electrónico |
| `password` | `password` | `Lock` | "••••••••" | Contraseña |
| `role?` | Select | `Shield` | "Selecciona un rol" | Rol |

Cada campo sigue el patrón establecido en `LoginCard` (ADR_04):
1. `<Label>` con clase `text-white/85`.
2. Contenedor `<div className="relative">` para posicionar el ícono y el input.
3. Ícono `lucide-react` con posicionamiento absoluto, `pointer-events-none`, `text-white/60`, centrado verticalmente con `top-1/2 -translate-y-1/2`.
4. Componente shadcn con clase `glass-input h-11 rounded-lg pl-9 pr-3 text-sm shadow-none`.

**Botón de envío**: `<Button type="submit">` con clase `glass-button h-11 w-full rounded-lg text-sm font-semibold tracking-wide shadow-none`, texto "Crear Usuario", sin estado `disabled` ni loading spinner (no hay comportamiento).

**Formulario sin comportamiento**: Por directriz explícita, el formulario no tiene:
- Estado local (`useState`)
- Manejadores de eventos (`onChange`, `onSubmit`)
- Validación de campos
- Lógica de envío o llamadas a API
- Integración con `useToast` o `sonner`

La etiqueta `<form>` se mantiene por razones semánticas y de accesibilidad, pero no tiene prop `onSubmit`.

### 2.4 Armonización visual del Select de shadcn con el glassmorphism

El `SelectTrigger` nativo de shadcn aplica `bg-transparent`, `rounded-lg`, `border border-input`, y altura `data-[size=default]:h-8` (32px). Para integrarlo visualmente con los `Input` del formulario se tomaron las siguientes decisiones:

1. **Aplicación de la clase `glass-input`**: Se añadió al `className` del `SelectTrigger`. Esto sobreescribe el fondo transparente con el efecto glass (`color-mix(in oklab, white 10%, transparent)`), el borde (`1px solid color-mix(in oklab, white 28%, transparent)`) y los estados hover/focus definidos en `index.css`. La clase `w-full` se agregó explícitamente porque el `SelectTrigger` base usa `w-fit`.

2. **Altura uniforme**: Se especificó `h-11` (44px) en el `className` para igualar la altura de los `Input`, anulando el `data-[size=default]:h-8` del componente. Se mantuvo `size="default"` para no perder otros atributos de tamaño.

3. **Ícono Shield con `z-10`**: A diferencia de los inputs nativos donde el ícono se pinta detrás del campo, el `SelectTrigger` de Radix renderiza su contenido sobre el ícono posicionado absolutamente. Se agregó `z-10` al ícono `Shield` para garantizar que quede visible por encima del trigger en todas las circunstancias.

4. **Placeholder**: El `SelectValue` recibe `placeholder="Selecciona un rol"`, que se muestra cuando no hay valor seleccionado. El estilo del placeholder está gobernado por `data-placeholder:text-muted-foreground` en el trigger base. Con `glass-input` aplicado y tema oscuro, el placeholder hereda un color tenue apropiado.

5. **Opciones**: `SelectItem` con valores `USER` y `ADMIN`, mostrando etiquetas en español ("Usuario", "Administrador"). Los valores corresponden exactamente a los literales del tipo `UserRole`.

## 4. Consideraciones y Consecuencias

### Positivas

- **Tipado centralizado y reusable**: La interfaz `User` y el tipo `UserRole` en `src/types/user.ts` pueden ser importados por cualquier componente futuro (tablas de usuarios, tarjetas de perfil, formularios de edición).
- **Consistencia visual total**: El formulario replica exactamente el lenguaje visual de `LoginCard` y `PostForm` (glass-card, glass-input, glass-button, íconos lucide-react, espaciado, tipografía), garantizando una experiencia de usuario uniforme.
- **Cero dependencias nuevas**: El componente Select de shadcn utiliza `radix-ui` que ya era dependencia del proyecto. No se agregaron paquetes nuevos al `package.json`.
- **Patrón de formulario extensible**: La estructura de campos con ícono + input/select posicionados es fácilmente replicable para formularios futuros (edición de usuario, creación de posts avanzados, etc.).
- **Preparado para iteración**: Al residir en `src/components/dashboard/`, el formulario está listo para que la siguiente iteración le agregue estado, validación con Zod y mutaciones con TanStack Query, sin cambios estructurales.

### Deuda técnica aceptada

- **Formulario sin comportamiento**: El componente es puramente presentacional. No se implementó estado, validación, ni integración con la API del backend. Esto se abordará en una iteración futura cuando el backend tenga los endpoints de usuarios completamente operativos.
- **Select sin valor controlado**: El `Select` de shadcn se renderiza en modo no controlado (sin prop `value`). Al no haber estado, el dropdown siempre muestra el placeholder. La versión final requerirá estado controlado (`value` + `onValueChange`).
- **Sin máscara ni validación de fecha**: El input `type="date"` delega la UX al navegador, sin picker personalizado ni validación de rango (ej. edad mínima).
- **Sin confirmación visual de contraseña**: No se incluyó un campo "Confirmar contraseña" ni el toggle de visibilidad que sí tiene `LoginCard`.
- **Sin responsive mobile**: Consistente con la decisión de ADR_04 (§4, Deuda técnica), el formulario está diseñado solo para escritorio.
